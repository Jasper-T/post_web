# 操作指南

本文档面向前端页面使用者，说明如何用 web-post 完成图片算法接口调试、批量测试、原始响应查看、解析结果查看、图片标注预览、GT 配置、结果评估、标注转换和可视化图片下载。

## 1. 启动和访问

web-post 可以作为 Windows/macOS/Linux 便携应用运行，也可以由管理员部署成 Web 服务后通过浏览器访问。

Windows 便携包通常是：

```text
web-post/
  web-post.exe
  start.bat
  web-post.env
  html/
  data/
  datasets/
```

Windows 启动方式：

1. 打开 `web-post` 文件夹。
2. 双击 `web-post.exe` 或 `start.bat`。
3. 浏览器通常会自动打开。
4. 如果没有自动打开，手动访问 `http://127.0.0.1:8000`。

macOS/Linux 便携包通常包含 `web-post` 可执行文件、`start.command` 或启动脚本、`web-post.env`、`html/`、`data/`、`datasets/`。启动方式以发布包内说明为准；如果从终端启动，一般是进入发布目录后运行：

```bash
./web-post
```

指定端口时，可在启动命令中添加 `--port`：

```powershell
.\web-post.exe --port 8001
```

```bash
./web-post --port 8001
```

如果 web-post 已部署为 Web 服务，直接在浏览器打开管理员提供的访问地址即可。此时不需要关心发布包目录，也不能通过本机修改 `web-post.env` 改变服务器配置。

不要单独移动可执行文件。请保持可执行文件、`html`、`data`、`datasets` 在同一个目录中。onedir 发布包里的 `_internal/` 也不能删除。

## 2. 文件范围和上传范围

- `Browse` 文件浏览器显示的是后端机器可访问的目录，根目录由 `WEB_POST_FILESYSTEM_ROOT` 控制。
- 上传文件时，浏览器从前端用户的机器选择文件。
- 上传目标固定是后端机器上的 `WEB_POST_DATASETS_ROOT`。

如果浏览器和后端不在同一台机器，`Browse` 看到的不是浏览器电脑的文件系统。需要把本机文件上传到后端后再选择。

### 2.1 修改文件浏览根目录

`WEB_POST_FILESYSTEM_ROOT` 决定 `Browse` 文件浏览器能看到的根目录。

桌面便携版推荐修改发布目录中的 `web-post.env`：

```text
# web-post.env
WEB_POST_FILESYSTEM_ROOT=D:\datasets
```

macOS/Linux 示例：

```text
# web-post.env
WEB_POST_FILESYSTEM_ROOT=/Users/yourname/datasets
```

修改后重新启动 web-post。Windows 便携版如果不配置，默认显示可执行文件所在磁盘的根目录，例如 `D:\`；macOS/Linux 默认通常是 `/`。

也可以启动时临时指定：

```powershell
.\web-post.exe --filesystem-root D:\datasets
```

```bash
./web-post --filesystem-root /Users/yourname/datasets
```

命令行参数只对本次启动生效，优先级高于 `web-post.env`。

如果使用的是已部署 Web 服务，`WEB_POST_FILESYSTEM_ROOT` 由服务管理员在服务器端配置。普通浏览器用户无法通过自己的电脑修改这个范围。

## 3. Pipeline 和模板管理

左侧 `Collections` 显示 Pipeline 分组和 Pipeline 列表。分组来自应用数据目录中的 `data/templates` 文件夹，`Ungrouped` 和 `Deleted` 是特殊分组，系统会自动保证存在。

每个 Pipeline 存在于：

```text
<应用数据目录>/data/templates/<group>/<pipeline_name>/
```

`pipeline_name` 是文件夹名，`displayName` 是配置文件里的显示字段。保存 Pipeline 时会校验两者全局唯一。

`Pipeline Name` 规则：

- 长度 3-64。
- 小写字母开头。
- 只能包含小写字母、数字、`_`、`-`。
- 不能以 `_` 或 `-` 结尾。
- 不能出现连续分隔符，例如 `__`、`--`、`_-`、`-_`。

### 3.1 复用其他用户的模板

默认发布包只包含 `Ungrouped/demo_image_pipeline`。真实模板往往包含私有接口地址、请求头、请求体、响应样例、类别名称或业务解析规则，因此不会作为默认公开模板一起发布。

复用模板时按以下方式操作：

1. 从其他用户处获取完整 Pipeline 文件夹，例如 `<pipeline_name>/`。
2. 将它复制到应用数据目录的 `data/templates/<分组>/<pipeline_name>/`。便携版通常是 `web-post/data/templates/<分组>/<pipeline_name>/`；已部署 Web 服务需要由服务管理员在服务器端复制。
3. 确认文件夹中至少包含 `pipeline.json`、`header.json`、`body.json`、`response.json`、`mapping.json`、`post_config.json`。
4. 如果复制后与已有 Pipeline 重名，修改文件夹名，并同步修改 `pipeline.json` 中的 `pipeline_name` 和 `displayName`。
5. 检查模板内是否带有私有 token、cookie、内网 URL 或敏感响应样例。对外分享前请先清理。
6. 刷新页面；如果左侧列表没有出现该模板，重启后端或重新启动便携版程序。

## 4. 配置 Request

选择 Pipeline 后，主工作区默认显示 `Request`。

顶部请求栏可配置请求方法、URL、Pipeline Name、Display Name 和 Image Path。`Image Path` 可以是单张图片，也可以是图片目录。点击 `Browse` 从后端文件系统选择路径，填写或选择后点击加载图片列表按钮，右侧 Response 图片列表会更新。

Request 视图包含：

1. `Header`：配置请求头 JSON。
2. `Body`：配置请求体 JSON。如果接口需要图片 base64，请预留字段，例如 `image_base64`、`image_b64`、`image`。
3. `Response`：保存接口响应样例 JSON，用于生成解析路径选项。
4. `Parsing`：配置如何从接口响应中提取检测结果。
5. `Request Config`：配置动态字段注入和超时时间。

## 5. 配置 Parsing

常用字段：

- `Collection Path`：检测结果列表路径，例如 `data.outputs`。
- `Item Path`：如果 Collection 内还有一层列表，在这里选择。
- `bbox`：数组形式 bbox，例如 `[x1, y1, x2, y2]`。
- `x1/y1/x2/y2`：坐标分散在多个字段时使用。
- `label`：类别名称。
- `class_id`：类别 ID。
- `conf`：置信度。
- `Extra Params`：额外字段。
- `Plot`：勾选后，该字段会显示在可视化标注文字中。

坐标模式：`BBox` / `Point`，以及 `XYXY` / `XYWH`。配置完成后点击 `Save`。

## 6. 发送图片和查看结果

1. 选择 Pipeline。
2. 确认 Header、Body、Response、Parsing、Request Config 已保存。
3. 设置 `Image Path` 并加载图片列表。
4. 切换到 `Response`。
5. 单张测试：点击图片行右侧发送按钮。
6. 批量测试：勾选图片后点击 `Send All`。

状态说明：

- `new`：未发送。
- `success`：请求成功，并已生成 raw/parsed 数据。
- `failed`：请求失败或解析失败。

`Response Result` 中：

- `Raw` 显示算法接口返回的原始 JSON。
- `Parsed` 显示经过 Parsing 规则转换后的标准检测结果。

如果 Raw 为空，请先确认右侧图片行状态是 `success`。如果 Raw 有内容但 Parsed 为空，通常是 Parsing 路径配置不匹配真实响应。

## 7. Image Preview 和 GT

`Image Preview` 用于查看标注图片。

- `Pred` 来自接口返回的 Parsed 结果。
- `GT` 来自真实标注文件。
- 点击刷新按钮可对当前图片重新生成 Pred 或 GT 标注图。
- 下载按钮用于下载标注图片文件夹，而不是保存单张结果。

配置 GT：

1. 点击右侧图片列表上方的设置图标。
2. 在 Settings 中设置 Format、Label Folder、Names。
3. Names 是当前图片和当前 Pipeline 预览、评估共用的类别列表。
4. 保存设置后，在 `Image Preview` 选择 `GT`。
5. 选择图片或点击刷新按钮生成 GT 标注图。

## 8. Cache Settings

点击右侧图片列表上方的设置图标打开 Settings。Settings 中包含当前缓存路径、cache folder 列表、刷新、全选、删除选中缓存，以及 GT 的 Format、Label Folder、Names 设置。

缓存目录结构：

```text
data/.cache/<pipeline>/<bucket>/
  post/
  parsed/
  pred/
  GT/
  labelme|voc|yolo/
```

bucket 命名示例：Image Path 为 `/root/1/2/3` 或 `/root/1/2/3/file.jpg` 时，bucket 都是 `2_3`。

## 9. Result Evaluation

`Result Evaluation` 用于把当前 Pipeline 的预测结果与 GT 标注对比。使用前需要：

- 至少成功运行过一张图片。
- `Parsed` 中有预测框。
- 已设置 GT Label Folder。
- 已设置 GT Format。
- 已设置 Names。
- 当前 Image Path 对应参与评估的图片或图片目录。

## 10. Annotation Conversion

`Annotation Conversion` 支持转换 Pred 或 GT。

Pred 转换使用当前已解析的预测结果。GT 转换使用当前 GT 标注目录。选择 `Target` 为 LabelMe、VOC 或 YOLO 后点击 `Convert`。

输出默认写入：

```text
data/.cache/<pipeline>/<bucket>/<target-format>/
```

## 11. 推荐完整流程

1. 新建 Pipeline。
2. 填写 URL、Pipeline Name、Display Name。
3. 配置并保存 Header。
4. 配置并保存 Body，确认图片字段存在。
5. 粘贴真实 Response 样例并保存。
6. 配置 Parsing。
7. 配置 Request Config，重点确认图片字段注入路径。
8. 设置 Image Path，加载图片列表。
9. 单张发送测试。
10. 查看 Raw、Parsed 和 Pred 标注图。
11. 如需 GT，打开 Settings 设置 GT Format、Label Folder、Names。
12. 批量勾选图片并点击 Send All。
13. 需要评估时进入 `Result Evaluation`。
14. 需要导出标注时进入 `Annotation Conversion`。

## 12. 常见问题

### Response Result 什么都不显示

先确认右侧当前图片状态是否为 `success`。状态还是 `new` 时 Raw/Parsed 都不会显示。

### Raw 有内容但 Parsed 为空

检查 Collection Path、bbox、label、class_id、conf 路径和坐标模式是否与真实响应一致。

### 点击 Send All 没有反应

检查 Pipeline 是否已保存、Image Path 是否已加载图片列表、右侧图片是否已勾选、URL/Body/Request Config 是否完整。

### 图片预览没有框

检查当前图片是否 success、Parsed 是否有 bbox、bbox 坐标是否超出图片范围、XYXY/XYWH 是否配置正确。

### 找不到本机文件

`Browse` 是后端文件系统，不是浏览器电脑文件系统。如果需要使用本机文件，请先上传到后端 datasets 目录。

### 页面还是旧版本

便携版确认访问的是当前启动程序地址，然后按 `Ctrl+F5`。已部署 Web 服务请联系管理员确认服务是否已更新。
